⚠️ Event Controller Stability Analysis Report

50 Website Simultaneous Event Distribution

Report Date: July 8, 2026
Project: Event Controller Plugin (Stetson Hills)
Analysis Scope: Scalability to 50+ remote website targets
Status: ⚠️ NOT RECOMMENDED for production without modifications

Executive Summary

Verdict: The current implementation is NOT STABLE for 50 websites.

  • Current Capability: 8-12 sites (reliable)
  • Attempted Capability: 50 sites
  • Primary Failure Mode: Request timeout after 30-60 seconds
  • Expected Success Rate at 50 sites: ~20% (only sites 1-10 complete)
  • Remaining Sites: Unknown state - orphaned processes

Critical Issues Identified

Issue #1: Sequential Processing Bottleneck ⏱️

Severity: CRITICAL

Current Flow:

Site 1 (wait 30s) → Site 2 (wait 30s) → Site 3 (wait 30s) ... → Site 50

Time Calculation:

  • 50 sites × 30 seconds timeout per site = 1,500 seconds (25 minutes)
  • PHP execution timeout default: 300 seconds (5 minutes)
  • Apache/Nginx request timeout: 30-60 seconds
  • Expected Result: ❌ Process terminates after ~10 sites

Code Location:

class-event-controller-form-handler.php (Lines 65-110)

foreach ( $selected_sites as $site_id ) {
    // ← BLOCKS on each iteration
    $result = $this->post_event_to_remote( $credentials, $event_data );
    // Waits up to 30 seconds for THIS site before moving to next
}

Impact:

Each site blocks the entire queue. If Site #3 is slow, Sites #4-50 wait idle. With 50 sites and realistic network conditions, the process will timeout before completing all submissions.

Issue #2: AJAX Request Timeout 🌐

Severity: CRITICAL

The Problem:

  • Browser AJAX waits for server response indefinitely
  • Browser/Server timeout: 30-60 seconds (varies by hosting)
  • After timeout: User sees "Request Failed" error
  • Meanwhile: Backend PHP still running in background
  • Result: Data becomes orphaned and inconsistent

Timeline Example (50 sites):

0:00   - Request sent to server
0:30   - ⏱️ AJAX timeout fires → User sees error message
5:00   - Backend finally finishes sites 1-50 (if it can)
End    - Sites 1-10 created, Sites 11-50 in unknown state
        User only sees generic error, no visibility into what succeeded

Code Location:

event-controller-form-script.php (Lines 280-290)

$.ajax({
  url: eventControllerData.ajax_url,
  type: 'POST',
  timeout: undefined,  // ← No explicit timeout, browser default kicks in
  success: handleSubmissionSuccess,
  error: handleSubmissionError
});

Issue #3: No Partial Results Recovery 🔄

Severity: CRITICAL

The Scenario:

When AJAX times out during 50-site submission:

  • Sites 1-10 may have been created successfully
  • Sites 11-50 in processing queue
  • User never learns what succeeded or failed
  • No way to identify which sites need retries
  • Frontend receives generic "Request failed" with no transaction details

Impact:

Manual audit required to determine what actually succeeded. Potential duplicate event creation if user retries.

Issue #4: Memory Growth Risk 💾

Severity: MODERATE

At 50 sites, response accumulation:

  • 50 responses × 10KB average = 500KB JSON payload
  • Plus error array and PHP object overhead
  • Shared hosting memory limit: often 64MB
  • Risk: Memory exhaustion on larger responses

Issue #5: Individual Timeouts Cause Cascade Failure 🔗

Severity: MODERATE

If any remote site delays response:

  • Site 1 slow (30s timeout) → blocks sites 2-50
  • No individual timeout + retry logic
  • All downstream sites affected by one slow server
  • No graceful degradation

Recommended Solutions

Option 1: Reduce Individual Timeouts (Quick Fix - Limited)

Implementation Time: 5 minutes | Effectiveness: 30% 

// Change from 30 seconds to 10 seconds per site
'timeout' => 10,  // ← Reduced

Impact:

  • 50 sites × 10s = 500 seconds (~8.3 minutes)
  • Still too long for typical browser timeout
  • Risk: Legitimate slow sites get cut off
  • Verdict: Band-aid solution, not recommended

Option 2: Background Job Queue (RECOMMENDED) ✅

Implementation Time: 2-3 hours | Effectiveness: 95%

Architecture:

User submits 50 sites
  ↓
Backend queues 50 jobs (instant)
  ↓
Returns to user immediately (5 seconds)
  ↓
Background processor handles jobs asynchronously
  ↓
Results stored in database
  ↓
User polls or gets webhook notification of completion

Implementation Approach:

  • Use WordPress Cron or external job queue (Laravel Queue, Bull, etc.)
  • Each background job processes one site
  • Store results in custom post meta or database table
  • Frontend polls for updates every 5 seconds
  • Total processing time: 50 × 30s in parallel = ~45-60 seconds background

Benefits:

  • ✅ User sees immediate confirmation (no 25-minute wait)
  • ✅ Background jobs process in parallel
  • ✅ Partial failures handled gracefully
  • ✅ Progress tracking possible
  • ✅ Can retry failed sites automatically
  • ✅ Scales to 100+ sites easily

Option 3: Batch Processing with Progress Updates

Implementation Time: 1-2 hours | Effectiveness: 80%

Architecture:

Request 1: Batch 1 (sites 1-10) → Wait 5s → Return progress
Request 2: Batch 2 (sites 11-20) → Wait 5s → Return progress
Request 3: Batch 3 (sites 21-30) → Wait 5s → Return progress
Request 4: Batch 4 (sites 31-40) → Wait 5s → Return progress
Request 5: Batch 5 (sites 41-50) → Wait 5s → Return progress

Benefits:

  • ✅ Each request stays under timeout
  • ✅ User sees real-time progress
  • ✅ Better UX feedback
  • ✅ Simpler to implement than Option 2

Drawback:

  • Total time still ~50 seconds (10s per batch × 5 batches)
  • More complex frontend logic

Option 4: Parallel Processing (Advanced)

Implementation Time: 4-6 hours | Effectiveness: 98%

Use cURL multi or Guzzle library for parallel HTTP requests:

$multi = curl_multi_init();
foreach ($selected_sites as $site_id) {
    $ch = curl_init();
    curl_setopt($ch, CURLOPT_URL, $remote_url);
    curl_multi_add_handle($multi, $ch);
}
// All requests execute in parallel, not sequentially
// 50 sites × 30s → ~45s max (if all slow)

Benefits:

  • ✅ All sites processed simultaneously
  • ✅ Time reduces from 1,500s to ~45s
  • ✅ Stays within browser timeout
  • ✅ Full visibility of all results

Complexity: Requires error handling, memory management

Testing Recommendations

Before Testing with 50 Sites, Test with Smaller Numbers:

  • 10 sites → should take ~5 minutes
  • 20 sites → should take ~10 minutes
  • 30 sites → likely fails with PHP timeout

Expected Breaking Point: Usually around 15-20 sites depending on hosting limits.

Test Scenarios to Run:

  • All sites respond quickly (best case)
  • One site very slow (30 second delay)
  • Multiple sites timeout (connection failures)
  • One site authentication fails
  • Monitor server logs for errors
  • Check database for orphaned/duplicate records

Conclusion

Current Status: The Event Controller plugin's multi-site distribution works well for up to 8-12 websites. Attempting to use it with 50 sites in its current form will result in timeout failures and data inconsistencies.

For 50-site support, implementation of one of the recommended solutions (preferably Option 2: Background Jobs) is mandatory. Without architectural changes, the system cannot reliably handle 50 simultaneous submissions.

Next Step: Test current limits with real data and document actual breaking point. Then prioritize implementation of batch processing or background jobs based on urgency.
